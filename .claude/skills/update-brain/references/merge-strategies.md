# Merge Strategies — File-by-File Reference

This file tells the update-brain skill how to handle each managed file in the vault.

## Classifications

| Class | Behavior |
|---|---|
| `system-code` | Code or scripts the user shouldn't edit. Replace with upstream by default; if user edited, show diff and offer "keep mine" override. |
| `section-merge` | Markdown files with `## ` section structure where users add custom protocols. Three-way merge at section level. |
| `user-data` | Files containing the user's personal content. Never auto-update — show upstream version and let user cherry-pick. |
| `additive-only` | Files that should be created if missing but never overwritten if present. |
| `runtime-state` | Generated by cron jobs / agent at runtime. Never touch. |
| `skip-heredoc-generated` | Templates for these exist in the repo but `lib/onboard.sh` generates the actual user files via inline heredocs. Until onboard.sh is refactored (v0.7), don't attempt to merge — the baseline doesn't match what the user has. |

## How to detect classification

For any file in the user's vault that matches a template:

1. Look up the file in this map.
2. If not in the map but exists under `06-Agent/`, `05-Meta/`, `07-Systems/`, or `08-CoreSystem/`: assume `user-data` (be conservative).
3. If in `00-Inbox/`, `01-Projects/`, `02-Areas/`, `03-Resources/`, `04-Archive/`: assume `runtime-state` — never touch.

## How to detect "user changed" vs "user unchanged"

Diff the user's file against the baseline (the templates at the version they originally installed). If `diff -q` reports no difference → user unchanged → safe to take upstream automatically.

For section-merge files, do this at the section level too — a user might have edited two sections and left the rest alone, allowing safe upstream-takes for the unchanged sections.
